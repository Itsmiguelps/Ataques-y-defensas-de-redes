### рҹ”ҙ 01 В· CDP DoS вҖ” DenegaciГіn de Servicio vГӯa Cisco Discovery Protocol

**Protocolo objetivo:** CDP (Cisco Discovery Protocol)

Inunda la tabla de vecinos CDP de un switch Cisco con cientos de entradas falsas construidas con Scapy. Cada paquete lleva TLVs aleatorios (`DeviceID: ROUTER-XXXXXX`) y se envГӯa a la direcciГіn multicast CDP `01:00:0C:CC:CC:CC`, causando agotamiento de CPU y memoria.

```bash
sudo python3 01_cdp_dos.py -i eth2 -c 1000 -d 0.001
```

| ParГЎmetro | DescripciГіn | Default |
|-----------|-------------|:-------:|
| `-i` | Interfaz de salida | вҖ” |
| `-c` | NГәmero de paquetes | `500` |
| `-d` | Delay entre paquetes (s) | `0.005` |
| `-v` | Verbose | off |

**Contramedida:** `no cdp run` global В· `no cdp enable` por interfaz

### рҹҹ  02 В· ARP MitM вҖ” Man-in-the-Middle por Envenenamiento ARP

**Protocolo objetivo:** ARP (RFC 826)

EnvГӯa ARP Replies no solicitados (Gratuitous ARP) a dos vГӯctimas cada segundo: le dice a cada una que la IP de la otra tiene la MAC de Kali. Con IP forwarding activo, Kali reenvГӯa el trГЎfico de forma transparente. Al detener el ataque (Ctrl+C), restaura las tablas ARP originales.

```bash
sudo python3 02_arp_mitm.py -t1 192.168.1.10 -t2 192.168.1.20 -i eth2
```

| ParГЎmetro | DescripciГіn |
|-----------|-------------|
| `-t1` | IP de la VГӯctima 1 |
| `-t2` | IP de la VГӯctima 2 |
| `-i` | Interfaz de ataque |

**Contramedida:** `ip arp inspection vlan 1` (DAI) В· `port-security maximum 1`

### рҹҹЎ 03 В· DHCP Spoofing вҖ” Servidor DHCP Falso (Rogue DHCP)

**Protocolo objetivo:** DHCP (RFC 2131)

Implementa un servidor DHCP fraudulento que responde al handshake DORA completo antes que el servidor legГӯtimo. Asigna como gateway la IP de Kali y un DNS controlado por el atacante, logrando MitM automГЎtico sobre todos los clientes que renueven su IP.

```bash
sudo python3 03_dhcp_spoofing.py -i eth2 \
  --pool 192.168.10.200-220 \
  --gateway 192.168.10.50 \
  --dns 8.8.8.8
```

| ParГЎmetro | DescripciГіn | Default |
|-----------|-------------|:-------:|
| `--pool` | Rango de IPs falsas | вҖ” |
| `--gateway` | Gateway anunciado (Kali) | вҖ” |
| `--dns` | DNS anunciado | вҖ” |
| `--lease` | Lease time (s) | `600` |

**Contramedida:** `ip dhcp snooping` В· `ip dhcp snooping trust` en uplink

### рҹҹ  04 В· DHCP Starvation вҖ” Agotamiento del Pool DHCP

**Protocolo objetivo:** DHCP (RFC 2131)

Genera MACs aleatorias con OUIs de fabricantes reales (VMware, VirtualBox, QEMU) y envГӯa DHCP Discovers masivos. En modo `--confirm`, completa el handshake DORA con cada MAC falsa, bloqueando definitivamente todas las IPs del pool hasta que expire el lease.

```bash
sudo python3 04_dhcp_starvation.py -i eth2 -c 500 --confirm --delay 0.1
```

| ParГЎmetro | DescripciГіn | Default |
|-----------|-------------|:-------:|
| `-c` | NГәmero de solicitudes | `254` |
| `--confirm` | Completar handshake DORA | off |
| `--delay` | Delay entre requests (s) | `0.05` |
| `--timeout` | Timeout por Offer (s) | `2` |

**Contramedida:** `switchport port-security maximum 2` В· `violation shutdown`

### рҹ”ө 05 В· MAC Flooding вҖ” Desbordamiento de la Tabla CAM

**Protocolo objetivo:** Ethernet 802.3

Genera tramas Ethernet con MACs de origen y destino completamente aleatorias en lotes de 100, usando mГәltiples hilos paralelos. La tabla CAM del switch (~8192 entradas) se satura rГЎpidamente: el switch no puede aprender MACs legГӯtimas y pasa a hacer flooding unicast a todos los puertos.

```bash
sudo python3 05_mac_flooding.py -i eth2 --continuous --threads 4
sudo python3 05_mac_flooding.py -i eth2 -c 50000 --vlan 10
```

| ParГЎmetro | DescripciГіn | Default |
|-----------|-------------|:-------:|
| `-c` | Tramas a enviar | `50000` |
| `--continuous` | Modo continuo | off |
| `--threads` | Hilos paralelos (mГЎx. 4) | `1` |
| `--vlan` | Tag 802.1Q opcional | вҖ” |

**Contramedida:** `port-security maximum 2` В· `mac-address sticky` В· `violation shutdown`

### рҹҹЈ 06 В· STP Root Claim вҖ” UsurpaciГіn del Root Bridge

**Protocolo objetivo:** STP 802.1D / PVST+ Cisco

EnvГӯa BPDUs de configuraciГіn STP/PVST+ con `Bridge Priority = 0` y `Root Path Cost = 0` cada 2 segundos a la direcciГіn multicast `01:80:C2:00:00:00`. Los switches, al recibir una prioridad menor a la propia (4096 > 0), inician la reelecciГіn del Root Bridge, reconvergiendo con Kali como nuevo Root y causando un DoS de 30вҖ“50 segundos.

```bash
sudo python3 06_stp_root_claim.py -i eth2 --priority 0 --hello 1 --vlan 10
```

| ParГЎmetro | DescripciГіn | Default |
|-----------|-------------|:-------:|
| `--priority` | Bridge Priority anunciada | `0` |
| `--hello` | Intervalo entre BPDUs (s) | `2` |
| `--vlan` | VLAN para PVST+ Cisco | вҖ” |
| `--duration` | DuraciГіn (0 = continuo) | `0` |

**Contramedida:** `spanning-tree bpduguard enable` В· `spanning-tree guard root`
